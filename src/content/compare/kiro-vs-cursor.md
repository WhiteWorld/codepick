## 核心理念差异

这是两者最本质的区别：

**Cursor** 的设计哲学是"**让你更快地写代码**"——更快的补全、更强的 Agent、更流畅的 Composer 多步骤编辑。它假设你知道自己要做什么，帮你提速。

**Kiro** 的设计哲学是"**帮你把想法变成生产级代码**"——通过 Spec（规格文档）驱动，先生成需求文档（requirements.md）、设计文档（design.md），再让 Agent 按规格执行。它假设你有个模糊的想法，帮你把它做完整。

---

## Kiro 的独特功能

### Specs（规格驱动开发）
Kiro 最大的差异化功能。当你描述一个功能需求，Kiro 会先生成：
1. **requirements.md**：用户故事 + 验收条件（EARS 格式）
2. **design.md**：技术架构 + 数据流 + TypeScript 接口定义

这些 Artifacts（产出物）让你在 AI 开始写代码前就能审查逻辑，大幅减少返工。

### Hooks（自动触发 Agent）
Kiro 支持在特定事件（保存文件、部署、测试失败）时自动触发 Agent 执行预设任务。例如：
- 每次保存 API 文件时，自动更新对应的文档
- 测试失败时，自动分析原因并提出修复方案

### Steering Rules
类似 Cursor 的 `.cursorrules`，但更结构化，支持按项目/文件类型分别配置 AI 行为准则。

---

## Cursor 的优势

### 更成熟的生态
Cursor 已运营多年，用户量更大，插件/扩展生态更完整，社区资源（教程、rules 模板）更丰富。

### 更多模型选择
Cursor 支持 Claude Opus 4.6、GPT-5、Gemini 2.5 Pro 等更多顶级模型；Kiro 目前主要用 Claude Sonnet 系列。

### 更快的代码补全
Cursor Tab 是业界公认最流畅的代码补全之一，Kiro 的补全体验目前不及 Cursor。

## 谁应该选哪个？

**选 Kiro，如果你：**
- 经常做"从零开始"的新功能开发
- 需要把模糊需求变成完整的实现方案
- 喜欢在写代码前有清晰的规格文档
- 想试用最新 AWS AI 产品（且现在免费）

**选 Cursor，如果你：**
- 已有成熟项目，需要高效迭代
- 对代码补全速度要求高
- 需要接入最新最强模型（Opus 4.6、GPT-5）
- 不想适应新工具的学习成本
